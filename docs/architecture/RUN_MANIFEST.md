# Run Manifest

**Purpose:** Every HC-GRC pipeline run emits a `manifest.json` at run close. The manifest is the single artifact that lets an external auditor, replicator, or the Adversarial Protocol arbiter answer "what exactly ran, on what data, with what agents, producing what outputs." The lab notebook captures intent and narrative. The manifest captures execution truth. Both are required; neither substitutes for the other.

**Authors:** Orchestrator emits at run close. Schema defined here. Provenance Agent validates.

**Location:** `runs/<run_id>/manifest.json` — DVC-versioned, linked to the lab notebook commit at run close.

## Field Reference

### Identity and Status

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `run_id` | UUID v4 | Yes | Generated by Orchestrator at run start |
| `run_type` | enum | Yes | `confirmatory` requires SAP lock and Gate 2 to have passed |
| `phase` | enum | Yes | Determines which data splits are accessible |
| `status` | enum | Yes | `halted-by-gate` means a human gate rejected; `halted-by-guardrail` means output failed Guardrails Agent |

### Topology

`topology_hash` is a SHA-256 of the parsed Handoffs DAG — the directed graph of "Receives from" / "Passes to" across all 48 AGENT.md cards at the `graph_commit` SHA. A topology hash change means the workflow changed. This is the primary identifier for "which version of the pipeline ran this."

### Seeds

`primary_seed` is derived from the RFC 3161 timestamp of the pre-registration commit (timestamp bytes → SHA-256 → first 8 hex digits → integer). Per-module seeds follow the rule `primary_seed + module_number` where module numbers are P1=1, P2=2, P3=3, P4=4, P5=5. The seed derivation method is documented here so it is independently verifiable.

### Data

`test_split_accessed` is a boolean written by the Data Steward agent. It is `false` for all exploratory runs. Any manifest where `run_type` is `exploratory` and `test_split_accessed` is `true` is a SAP violation and must be flagged to the Orchestrator and logged as an incident.

### Intermediate Edits

Any agent that modifies an artifact (not just produces it) must emit an entry to `intermediate_edits`. This field addresses the Beel et al. 2025 finding that undocumented LLM-based intermediate edits were the leading cause of reproducibility failure in the Sakana AI Scientist platform. An empty `intermediate_edits` array means no agent modified any artifact mid-run — it does not mean the field is optional.

### Gate Decisions

`dissent` is a required field. `null` is a valid value meaning no dissent. An array with no dissent entries is suspicious; the pre-registration ledger cross-references gate decisions for pattern analysis. Every gate rejection must be in `rejected` status with a non-null rationale. Gate decisions with rationale under 20 words are flagged at schema validation.

## Validation Rules

1. `run_type == "confirmatory"` requires `pre_registration.sap_commit_hash` to be non-null.
2. `run_type == "confirmatory"` requires `data.test_split_accessed == true`.
3. `run_type == "exploratory"` requires `data.test_split_accessed == false`.
4. Every agent with `status == "primary"` in the agent roster must appear in `agent_revisions`.
5. All `gate_decisions[].dissent` fields must be present (null is valid; absent is not).
6. `seeds.primary_seed` must match the derivation from `pre_registration.rfc3161_timestamp` when that timestamp exists.
7. `intermediate_edits` must be present (empty array is valid; absent field is not).
